# AGENTS Rules（项目宪法）

## A. TODO-AI 强约束（必须执行）
1. 所有写着 `// TODO-AI:` 开头的注释，都是需要 AI 实现的，看到就必须立即分析并处理，不能忽略。完成后才能改成 `DONE-AI` 状态。
2. 每个任务完成后，都必须在项目中搜索 `TODO-AI:`，确认是否还有未完成任务。
3. 开发过程中一旦发现 `TODO-AI:`，必须立即纳入当前任务处理，不能跳过。
4. 用户可能会同步改代码。若出现改动冲突、实现失败，或开始新任务时，必须重新阅读相关代码后再继续。
5. 当被要求进行 MCP 测试时，必须真正解决问题后才能进入下一步。项目依赖真机物联网设备，不能用模拟器替代验收。

## B. 设计与实现原则（新增：减法优先）
1. **减法优先于加法**：
   - 在实现新需求、修 bug、重构前，先做删除与收敛评估：
   - 哪些状态是冗余的？
   - 哪些分支是重复的？
   - 哪些代码可以直接删除而不损失能力？
   - 评估完成后，再决定是否新增代码。
2. **计算属性优先于可变存储**：
   - 能由已有状态推导出的值，默认使用计算属性，不新增存储字段。
   - 除非有明确性能或生命周期理由，否则不引入可变冗余状态。
3. **语法级可靠性优先**：
   - 优先使用 Swift 语言特性保证一致性（计算属性、不可变常量、类型约束），减少依赖“人工约定”的同步逻辑。
4. **每次提交都要回答两件事**：
   - 这次删掉了什么？为什么可以删？
   - 这次新增的状态里，哪些本可用计算属性替代？为何没有替代？

## C. Monorepo 目录约定（新增）
1. iOS 工程统一位于 `ios`。
2. 后端 TypeScript 代码统一位于 `backend`。
3. 前端子项目位于 `mishuAppUI`（子模块）。
4. 任何脚本、文档、规则、命令示例都应使用 monorepo 新路径，避免引用仓库根目录旧路径。

## D. UI 对齐规则（重要）
1. `mishuAppUI` 是为 AI 准备的 Web 原型与 UI 基准。
2. 未经用户明确要求，不要修改 `mishuAppUI` 里的 `.tsx` 原型代码来“适配 iOS”。
3. 当用户说 iOS UI 不对时，默认应阅读并参考 `mishuAppUI` 对应页面/组件的 `.tsx`，然后修改 `ios` 下的 SwiftUI 实现，让真机 iOS App 尽量和 Web 原型保持一致。
4. `imageForIosApp` 是 iOS 可用图片素材来源；需要补齐或替换 iOS 图片时，优先用该目录素材同步到 `ios/Resources/Assets.xcassets`，而不是改 Web 原型。
5. 可以运行 `mishuAppUI` 原型并用手机视口截图做视觉参照，但参照结果应服务于 iOS 代码修改。
6. 做 Web 原型手机截图时，必须使用真正的移动端视口/MCP/浏览器设备模拟；不要把普通 headless `--window-size` 截图当成手机渲染结论。

## E. 本地运行与调试常识
1. 后台/隧道通常由用户先启动：在仓库根目录运行 `./startFrpc.sh`，并在 `backend` 目录运行 `bun run dev`。
2. 如需自行启动后端，应使用 monorepo 路径：`cd backend && bun run dev`。
3. 模拟器可用于普通登录流程验证，测试账号为 `15680069020`，密码为 `111111`。
4. 涉及物联网设备能力或用户明确要求 MCP 真机测试时，仍必须用真机设备验证，不能用模拟器替代最终验收。

## F. 执行口径
- 不以“功能可跑”作为完成标准，必须同时满足：
  - 功能正确
  - 结构可维护
  - 冗余受控（优先减法）
  - 推导值尽量计算属性化
